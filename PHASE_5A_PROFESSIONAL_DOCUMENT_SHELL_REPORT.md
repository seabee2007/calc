### `DocumentStatusBadge`

| Prop | Type | Required |
|------|------|----------|
| `status` | `string` | Yes |

**Named statuses:** Draft, Submitted, Under Review, Approved, Approved as Noted, Revise and Resubmit, Rejected, Closed, Accepted, Declined.

Key normalization: lowercase + strip spaces/underscores/hyphens — so `"under_review"`, `"Under Review"`, and `"UNDER_REVIEW"` all resolve to the same color.

Unknown status strings fall back to a neutral slate pill using the original string as label.

## Style rules

- **White paper surface:** Root element is always `bg-white text-slate-950 print:bg-white print:text-black`. No `dark:` variants inside document body — identical to the Change Order document pattern, which renders correctly in both light and dark app themes.
- **Typography:** `text-xs font-semibold uppercase tracking-wide text-slate-500` for labels; `text-sm text-slate-900` for body values.
- **Info rows:** `border-b border-slate-100 py-2 last:border-0` — flex row with `sm:w-36` label column, matching Change Order `InfoRow`.
- **Placeholder handling:** All values pass through `displayValue()` from `previewDisplay.ts`. Empty/null/`[token]` values render as `—`. No raw `undefined` or template tokens reach the DOM.

## Naming rationale: `PaperDocumentSection` vs `DocumentSection`

The engine exports `DocumentSection` from `src/features/documents/types.ts` — it is a clause data type (`{ clauseKey, title, body: string }`). The new UI wrapper is named `PaperDocumentSection` to prevent import confusion; TypeScript consumers can import both without collision.
